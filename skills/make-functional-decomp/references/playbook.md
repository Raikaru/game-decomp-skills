# Functional Decomp Playbook

## Repo Skeleton

- `.gitignore`: ignore user-owned artifacts, private fixtures, generated copyrighted content, traces, and build outputs.
- `README.md`: quickstart, artifact requirements, test commands, compatibility scope, and contribution expectations.
us|- `docs/provenance.md`: allowed sources, observation note rules, symbol/type confidence, and policy.
- `docs/compatibility.md`: target versions, supported content, file formats, saves, scripts, networking, demos/replays, mods, and known bugs.
- `docs/deviations.md`: known differences from original behavior and whether they are intentional.
- `docs/oracles.md` or `docs/behavior-oracles.md`: how behavior is observed, compared, and accepted.
- `src/`: readable reconstructed implementation.
- `include/`: reconstructed APIs, types, constants, and confidence notes where helpful.
- `src/platform/`: OS, filesystem, input, timing, audio, rendering, and threading boundaries.
- `src/formats/`: parsers for original data formats; no copyrighted inputs.
- `tests/`: behavior tests, save/script round-trips, parser tests, and state snapshots.
- `tests/fixtures/generated/`: synthetic fixtures generated by scripts.
- `tests/fixtures-private/`: ignored local user-owned fixtures.
- `tools/observe/`: local observation, tracing, or comparison helpers.
- `tools/generate-fixtures/`: synthetic fixture generation.
- `tools/extract/` and `tools/convert/`: local-only extraction/conversion helpers.
- `docs/notes/`: observations and hypotheses; avoid copied implementation.
- `userdata/`, `original-private/`, or `assets-private/`: local ignored paths for user-owned artifacts.

## First Milestone

1. Define the target version and artifact hash externally.
2. Reconstruct one small, observable subsystem.
3. Add a behavior test or trace comparison for it.
4. Add a local-only path for user-owned comparison data.
5. Document known deviations and unknowns.

Expected deliverables:

- `README.md` quickstart.
- `.gitignore` with private artifact and fixture exclusions.
- `docs/provenance.md`.
- `docs/compatibility.md`.
- `docs/behavior-oracles.md`.
- `docs/deviations.md`.
- One synthetic fixture generator.
- One behavior test.
- One documented private fixture path.

rr|## Provenance Practice

- Store observations as behavior, not copied implementation.
- Prefer "what happens" notes over "the original code does this" notes.
- Use temporary descriptive names when symbol provenance is uncertain.
- Review commits for accidental copied strings, tables, assets, or proprietary references.
- Record source, URL, license, and confidence for public symbols, signatures, headers, or documentation.
- Do not scrape decomp.me or similar collaborative services; use documented APIs where permitted or user-provided scratch links/content.

## Behavior Test Patterns

- Parser tests with synthetic binary fixtures.
- Save encode/decode round-trips with generated test saves.
- Script VM opcode tests and scheduling tests.
- Deterministic state snapshots after fixed input sequences.
- Audio/event timing tolerances, not only "plays without crashing."
- Content smoke tests that run locally against user-owned data.

## Tool Selection

- Use Ghidra/IDA/radare2 or architecture-specific disassemblers for local observation, not as final-source authorities.
- Use m2c or similar decompilers as rough scaffolding; rewrite into maintainable source and record provenance.
- Use trace/log comparison tools for behavior oracles when exact assembly matching is not the goal.
- Use fuzzing or property tests for parsers and data-format loaders.
- Use generated fixtures for committed tests and user-owned fixtures for local full-content tests.

## Non-Obvious Failure Modes

- Readability refactors can erase quirks that content, scripts, or saves depend on.
- Decompiler-shaped source can fossilize wrong types and control flow.
- Synthetic tests can pass while real content fails on obscure format variants.
- Save compatibility can become impossible if internal schemas are not versioned early.
- Symbol names from unclear sources can contaminate provenance.

## Review Prompts

- Are original artifacts and copyrighted fixtures excluded from the repo?
- Is behavior captured in tests or only in prose?
- Are names/types/constants marked when speculative?
- Do stubs fail loudly and explain missing behavior?
- Are platform dependencies isolated enough for a future PC port?
